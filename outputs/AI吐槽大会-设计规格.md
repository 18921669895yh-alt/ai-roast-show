# AI 吐槽大会：产品与技术设计规格

日期：2026-07-13  
状态：待用户最终审核

## 1. 产品目标

「AI 吐槽大会」是一个无需登录即可体验的互动喜剧网站。用户上传照片，或粘贴朋友圈文案、聊天记录、自我介绍等文字素材；AI 根据用户主动提供的细节生成有观察力、有画面感、有反差但不羞辱人的脱口秀式吐槽。用户可以对 AI 进行最多五轮即兴反击，最后生成可分享的喜剧观察报告与竖版分享卡。

核心体验不是“使用一个 AI 工具”，而是“参加一档可以玩的深夜喜剧节目”。成功标准是用户觉得“说得太准了”，而不是“被攻击了”。

## 2. 已确认范围

### 2.1 本期包含

- 首页、素材输入、生成结果、即兴对决、吐槽报告五个主要页面。
- 分享卡片预览弹窗与隐私/内容原则弹窗。
- 图片上传、拖拽、预览、格式与大小校验。
- 文字输入与素材类型快捷选择。
- 三档吐槽浓度，默认“熟人互损”。
- 真实 DeepSeek 与 Kimi API 接入，以及无密钥或服务异常时的 Mock 演示模式。
- 图片固定使用 Kimi 多模态模型；纯文字优先 DeepSeek，失败时回退 Kimi。
- 最多五轮即兴反击。
- 报告生成、本场最佳、虚构指标、奖项与 PNG 分享卡导出。
- 本地保存最近一次文字结果和报告数据，不持久化照片。
- 响应式布局、深色主题、键盘操作与 `prefers-reduced-motion`。
- 适量生成式视觉资产：舞台背景、CRT 外壳纹理、节目贴纸、奖杯或麦克风装饰。

### 2.2 本期不包含

- 用户账号、注册、登录、云端历史记录。
- 公开广场、排行榜、关注、评论或多人实时互动。
- 付费、订阅、额度购买。
- 用户照片服务器存储、数据库或对象存储。
- 真实心理测量、人格诊断或年龄/身份推断。

## 3. 视觉方向

### 3.1 核心风格

采用“复古电视台 × 深夜脱口秀”的节目包装。主色为深紫黑、米白、荧光黄、热烈红和紫色。视觉语言包括 CRT 电视框、频道编号、扫描线、直播灯牌、字幕条、评分票、票根、贴纸、手写圈画、麦克风和观众笑声波形。

首页使用完整 CRT 舞台形成记忆点；其余页面只保留节目包装语言，避免所有内容都被厚重电视外框包住。扫描线、颗粒和噪点仅用于背景，不覆盖正文。

颜色职责：

- 黄色：主要行动、频道选中、关键标题。
- 红色：直播状态、反击、警示与高能包袱。
- 紫色：AI 选手、特殊节目与辅助强调。
- 米白：正文、报告纸张和长文本阅读面。

中文标题使用粗黑高冲击字体；正文使用清晰的中文无衬线字体；数字评分使用等宽字体。生成式图片不承载关键文字、按钮或数据。

### 3.2 页面视觉分工

- 首页 / CH 01：完整 CRT 舞台、频道灯牌、滚动字幕和观众席。
- 素材页 / CH 02：素材带、机位标签和舞台字幕，操作区域保持清晰。
- 结果页 / LIVE：逐句字幕、观众反应、评分牌和“今晚最好笑的一句”。
- 对决页 / VS：左右选手牌、回合牌、拳击铃视觉；手机端改为上下布局。
- 报告页 / END：像从电视里打印出的米白档案纸，适合长内容和分享导出。

## 4. 信息架构与路由

- `/`：首页，介绍产品、示例、三步流程、吐槽浓度和底部 CTA。
- `/roast`：素材输入、图片预览、文字输入、模式与浓度选择、生成状态。
- `/result`：主持人开场、3–5 条细节吐槽、本场最佳、反向夸奖与观众反应。
- `/battle`：最多五轮即兴对决、快捷反击、回合评分与最终判定。
- `/report`：人格标签、特征依据、虚构数据、年度奖项、本场最佳与分享入口。
- 全局弹窗：分享卡预览、隐私与内容原则。

导航中的“开始吐槽”“吐槽报告”“即兴对决”“关于节目”分别指向对应页面或说明区。如果缺少最近一次结果，报告和对决入口显示节目化空状态并引导先完成一次吐槽。

## 5. 页面与核心组件

### 5.1 首页

`Header`、`HeroSection`、`SpotlightBackground`、示例吐槽卡、三步票根流程、`RoastLevelSelector`、底部 CTA 与 `Footer`。

Hero 主标题为“来都来了，让 AI 说你两句”，主要按钮进入 `/roast`，次要按钮滚动到“今日观众席”。Logo 连点五次触发“AI 自我吐槽大会”彩蛋。

### 5.2 素材与生成

`RoastUploader` 支持 JPG、PNG、WEBP，默认上限 10MB；展示预览、替换与删除。`TextMaterialInput` 支持文字素材。至少提供图片或文字之一；同时提供时优先视为多模态任务并走 Kimi。

`RoastModeSelector` 选择照片、穿搭、朋友圈、聊天风格、自我介绍或随机开麦。`RoastLevelSelector` 提供 30%、65%、90% 三档；90% 档需要轻量确认，但安全规则不随强度降低。

`LoadingComedyStage` 在请求期间轮播节目化文案。API 完成后再由前端按喜剧节奏逐句揭晓，避免结构化 JSON 流式解析的不稳定性。

### 5.3 结果页

`RoastResultCard` 展示观察标题、吐槽正文和标签。`BestJokeCard` 突出本场最佳。`AudienceReactionBar` 支持“哈哈哈”“有点准”“太损了”“不服”“申请反击”。反应只保存在当前浏览器。

“不服”与“申请反击”进入 `/battle`；“保存为分享卡片”打开 `SharePoster` 弹窗；“领取吐槽报告”进入 `/report`。

### 5.4 即兴对决

`ComebackBattle` 保存当前五轮以内的对话上下文。每轮用户可输入自定义反击或使用快捷句。AI 必须先接住用户的梗，再给出不贬低用户的反转。

`BattleScoreboard` 展示接梗能力、反击力度、嘴硬指数和观众支持率。第五轮后从“用户获胜”“AI 险胜”“双方嘴都很硬”“本场无人认输”中给出娱乐化结果。

### 5.5 报告与分享

`RoastReport` 组合三个喜剧标签、最有趣特征、虚构数据、年度奖项与最佳金句。`ComedyMetricCard`、`AwardCard` 和 `SharePoster` 负责可复用展示。

所有指标旁明确标注“以下数据均为喜剧化虚构，不是真实心理测量”。分享卡采用竖版社交媒体比例，在浏览器中渲染并导出 PNG，包含 Logo、素材摘要、标签、最佳金句、嘴硬指数和网址/二维码预留位置。

## 6. 系统架构

使用 Next.js App Router、React、TypeScript、Tailwind CSS、Framer Motion 与 Lucide Icons。核心分层：

1. 页面与组件层：展示、交互和动画。
2. 客户端状态层：当前素材、生成状态、最近结果、对决上下文和弹窗状态。
3. API 路由层：输入校验、模型路由、超时、重试、回退和错误标准化。
4. 模型适配层：统一 DeepSeek 与 Kimi 的请求/响应接口。
5. 安全与结构层：提示词安全边界、Zod 结构校验、输出过滤与降级。

建议 API：

- `POST /api/roast`：生成主持人开场、细节吐槽、本场最佳、反向夸奖和报告种子。
- `POST /api/comeback`：基于当前回合上下文生成接梗反击和评分。
- `POST /api/report`：在需要时补全报告标签、特征和奖项；已有结果可作为输入。

服务端环境变量：

- `DEEPSEEK_API_KEY`
- `DEEPSEEK_BASE_URL`，默认 `https://api.deepseek.com`
- `DEEPSEEK_MODEL`，默认使用当前非思考快速模型配置
- `MOONSHOT_API_KEY`
- `MOONSHOT_BASE_URL`
- `KIMI_MODEL`，默认使用支持图片理解的 Kimi 多模态模型配置
- `AI_MOCK_MODE`，可强制启用 Mock

模型名通过环境变量配置，避免供应商升级或弃用时修改业务代码。

## 7. 模型路由与数据流

### 7.1 路由规则

- 包含图片：使用 Kimi 多模态模型。
- 只有文字：优先 DeepSeek。
- DeepSeek 超时、限流、服务错误或输出结构两次校验失败：回退 Kimi。
- Kimi 图片请求失败：保留本地图片预览并提示重试；若同时有文字，可询问用户是否仅基于文字继续。
- 缺少所需密钥或 `AI_MOCK_MODE=true`：使用 Mock 适配器。

### 7.2 图片隐私

图片保留在浏览器内存中，仅在用户点击生成后通过 HTTPS 提交到本项目的服务端路由，再临时转发给 Kimi。服务端不写入磁盘、日志、缓存、数据库或对象存储。响应结束后不保留图片数据。浏览器本地持久化仅保存文字结果、报告数据和非敏感配置，不保存图片或 Data URL。

### 7.3 结构化响应

`RoastResult` 至少包含：

- `opening`
- `observations[3..5]`：`title`、`body`、`tag`
- `bestJoke`
- `reverseCompliment`
- `comedyTags[3]`
- `metrics`
- `award`
- `safetyMode`

服务端使用 Zod 校验。首次格式失败时进行一次 JSON 修复或重新请求；仍失败则使用安全 Mock 结果并返回可识别的降级标记。

## 8. 内容安全

系统提示与输出检查共同执行以下规则：

- 不进行容貌、身材或不可改变身体特征的羞辱。
- 不涉及种族、宗教、性别、性取向、残障、疾病等敏感身份。
- 不推测收入、职业、心理疾病、家庭状况或生活经历。
- 不进行人格诊断，不使用恶毒侮辱，不鼓励霸凌。
- 不对未成年人进行尖锐吐槽。
- 虚构指标必须明确标注娱乐性质。
- 检测到明显低落、自我否定或危机表达时，切换温和模式，以支持性反向吐槽收尾。

允许观察的对象仅限用户主动提供的穿搭选择、拍照姿势、文字表达、Emoji 使用、朋友圈风格、聊天语气、自我介绍方式和生活小习惯。

## 9. 状态、错误与降级

- 空状态：说明需要提供的素材，并给出可点击示例。
- 上传错误：在上传区显示具体原因，不清空已有文字。
- 加载状态：节目化轮播文案，允许取消当前请求。
- API 超时/限流：自动回退；最终失败时保留输入并提供“重新开麦”。
- 结构错误：自动修复一次，再降级安全 Mock。
- 无历史结果：报告与对决页面显示引导入口。
- 本地存储损坏：忽略无效数据并恢复空状态，不阻塞页面。
- PNG 导出失败：保留预览，提供复制金句与再次导出。

## 10. 动画与无障碍

- 标题分段登场、聚光灯缓慢移动、吐槽逐句出现、包袱轻微震动。
- “哈哈哈”飘出笑声文字，“不服”短暂变红，评分数字快速滚动。
- 报告像打印纸输出，分享卡像幕布拉开，回合切换显示拳击铃视觉。
- 动画轻快且不阻塞阅读；在 `prefers-reduced-motion` 下禁用持续移动、震动和滚动数字。
- 所有按钮有可见焦点、语义标签和至少 44px 的移动端触控区域。
- 图片预览提供替代文本；颜色不作为唯一状态信号。
- 手机端无横向滚动，正文不使用过小字号。

## 11. 生成式视觉资产策略

使用图像生成制作少量高质量装饰资产：舞台背景、CRT 电视外壳纹理、贴纸组、奖杯/麦克风装饰。资产需经过人工视觉检查和压缩，提供适合不同分辨率的尺寸，并设置纯色或 CSS 降级背景。

标题、正文、按钮、图表、分数和状态提示全部由 HTML/CSS 渲染，不嵌入生成图片。这样保证中文准确、可访问、响应式和可维护。

## 12. 测试与验收

### 12.1 单元测试

- 模型路由和回退优先级。
- 图片类型、大小和空素材校验。
- Zod 结构校验与修复降级。
- 安全模式切换和敏感输出拦截。
- 最近结果的本地存储序列化与损坏恢复。
- 对决五轮上限与评分范围。

### 12.2 API 集成测试

- DeepSeek 文字成功、超时、限流、格式异常和 Kimi 回退。
- Kimi 图片成功、失败与仅文字继续。
- 无密钥和强制 Mock 模式。
- 错误响应不泄露 API 密钥、完整提示词或图片数据。

### 12.3 浏览器流程测试

- 文字素材完成吐槽、查看结果、进入报告并导出 PNG。
- 图片上传、预览、替换、删除和生成。
- 三档浓度与 90% 确认。
- 观众反应、最多五轮对决和结算。
- 无历史结果、API 失败和重试。

### 12.4 视觉与可访问验收

- 常见手机、平板和桌面宽度下无横向滚动。
- 复古电视包装不降低正文对比度和可读性。
- 键盘可完成主要流程，焦点顺序合理。
- 减少动态效果生效。
- PNG 分享卡内容完整、中文清晰、比例适合手机社交媒体。

## 13. 交付标准

- 可安装、可启动、可构建的 Next.js 项目。
- `.env.example` 说明 DeepSeek、Kimi 与 Mock 配置，不包含真实密钥。
- 真实 API 与 Mock 模式均能完成核心流程。
- 代码按页面、组件、模型适配、安全、类型和测试拆分。
- README 包含本地运行、配置密钥、隐私行为与部署注意事项。
- 完成构建、自动化测试、关键浏览器流程和视觉检查后再交付。
